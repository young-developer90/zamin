ホーム/パフォーマンス

パフォーマンス

Zamin は速度を重視して設計されています。このページでは、VM アーキテクチャ、ベンチマーク比較、最適化手法、効率的な Zamin コードを書くための実践的なヒントを紹介します。

概要

Zamin のランタイムは、次の 3 つの主要コンポーネントで構成されています。

  • バイトコードコンパイル - ソースコードは実行前にコンパクトなバイトコードにコンパイルされます。コンパイラはコンパイル時に定数畳み込み、デッドコード除去、ピープホール最適化を実行します。
  • レジスタベース VM - 仮想マシンはレジスタベースの命令セットを使用してバイトコードを実行し、スタックベースの設計と比較してメモリトラフィックを削減します。
  • マーク＆スイープガベージコレクタ - 世代別、非移動型の GC で、インクリメンタルなコレクションフェーズによりポーズ時間を最小限に抑えながら、メモリオーバーヘッドを低く抑えます。

Python 3 とのベンチマーク比較

同じマシン上で Zamin と Python 3 を比較した実用的なベンチマーク。計測には time を使用（ウォールクロック計測）。値が小さいほど良い。

ベンチマークZaminPython 3比率
再帰 fib(32)1.51s0.35s4.3x
整数ループ (500万回)0.47s0.53s0.9x
文字列連結 (10万回)3.14s0.23s13.4x
リスト追加 (50万回)0.13s0.09s1.5x

Zamin はタイトな整数ループで優れた性能を発揮し、多くの場合 CPython に匹敵またはそれを上回りますが、バイトコードインタプリタのオーバーヘッドにより文字列操作やリスト操作では低速です。

最適化の影響

ピープホールオプティマイザと特殊化された整数オペコードは、コア VM プリミティブに大幅な高速化をもたらします。

最適化影響
Raw byte dispatch関数ポインタテーブルと比較して命令ディスパッチが 15～25% 高速化
特殊化された整数オペコードホットな整数ループでのボックス化と型チェックを回避
定数畳み込み定数式のランタイム評価を排除
Jump threading無条件ジャンプの連鎖を短縮
デッドコード除去return/throw 後の到達不可能な分岐を削除

最適化手法

Raw Byte Dispatch

VM メインループは、関数ポインタディスパッチテーブルの代わりに raw byte dispatch（GCC/Clang では computed gotos、それ以外では switch ベースのフォールバック）を使用します。これにより命令ごとに間接呼び出しが排除され、すべてのワークロードで一貫して 15～25% の改善が得られます。

特殊化された整数オペコード

頻繁に使用される整数演算には、汎用オペランドディスパッチを回避する専用のオペコードがあります。例えば OP_ADD_INT_IMM はボックス化や型チェックを行わずに整数レジスタに小さな定数を加算し、OP_LOAD_INT_CONST はコンパイル時定数をレジスタに直接ロードします。

ピープホールオプティマイザ

バイトコードコンパイラはピープホールオプティマイザを実行し、次の処理を行います。

  • 定数畳み込み - 定数式のコンパイル時評価
  • デッドコード除去 - 到達不可能な分岐と未使用の代入の削除
  • 強度低減 - 高コストな操作をより安価な同等処理に置き換え（例：x * 2 を x << 1 に）
  • Jump chaining - 無条件ジャンプの連鎖を単一のターゲットに短縮
  • オペコードマージ - load と store のシーケンスを単一の fused 操作に結合

パフォーマンスのヒント

  • 連結には f-string を使用 - "Hello {name}" は "Hello " + name よりも高速です。中間的な文字列の割り当てを回避できるためです。
  • リストを事前に割り当て - 最終的なサイズがわかっている場合は list.new(capacity) を使用して、繰り返しの再割り当てを回避します。
  • ローカル変数を優先 - VM ではローカル変数へのアクセスはグローバル変数へのアクセスより約 10 倍高速です。
  • インデックスループよりもイテレータを使用 - イテレータベースのループは各反復での境界チェックを回避します。
  • ループ内での繰り返し割り当てを回避 - 不変の計算はループ本体の外に移動します。文字列やリストは一度作成して再利用します。
  • 本番環境では zamin build を使用 - スタンドアロンバイナリをビルドすると、インタプリタでは実行できない追加の AOT 最適化が有効になります。
  • 最適化の前にプロファイリング - zamin run --disassemble を使用してバイトコードを検査し、ホットスポットを特定します。
